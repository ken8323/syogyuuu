## Git規約

### ブランチ命名

- `feature/<name>` — 新機能
- `fix/<name>` — バグ修正
- `chore/<name>` — 保守・設定変更
- `test/<name>` — テスト追加のみ

### コミットメッセージ

Conventional Commits形式:

- `feat:` 新機能
- `fix:` バグ修正
- `chore:` 保守
- `test:` テスト追加
- `docs:` ドキュメント変更

### ワークフロー

#### 要望・バグ報告を受けた時のプロセス（必須）

> ⚠️ 新機能・改善要望だけでなく、**バグ報告・不具合指摘**にも同じプロセスを適用すること。「軽微な修正だから」と判断して直接修正・pushしない。

1. `gh issue list` で既存Issueに該当するものがないか確認する
2. 該当Issueがなければ、新しいIssueを作成する
3. **Issue作成後、すぐに実装しない** — 人間がどのIssueに取り掛かるか選択するのを待つ
4. 優先順位の判断は人間が行う。勝手に実装を始めない

#### Issueの実装指示を受けた時のプロセス（必須・順序厳守）

> ⚠️ 手順1→2→3→4→5→6→7は必ず順番に実施すること。調査前にヒアリングしない。設計書更新前に実装しない。

1. `/sync-issue <Issue番号>` で作業開始を記録する（in-progressラベル＋コメント）
2. `gh issue view <Issue番号>` でIssue内容を再確認する（記憶に頼らない）
3. **調査** — 以下を全て実施してから次の手順に進む
   - 関連する `docs/` 配下の設計書を読む（Issue が参照するセクションを確認）
   - 影響を受ける既存ファイルを特定し、実際にコードを読む
   - `src/` のディレクトリ構成を確認し、新規作成ファイルの置き場所を把握する
4. **ヒアリング** — 以下の4観点を必ず全てチェックし、1つでも該当すればユーザーに確認する
   - [ ] 実装方針が複数考えられるか → 選択肢を提示して選んでもらう
   - [ ] Issueの記述に曖昧な点や不足があるか → 具体的に質問する
   - [ ] 設計書とIssueの間に矛盾があるか → どちらを優先するか確認する
   - [ ] 想定外の影響範囲（スコープ外ファイルへの変更等）があるか → スコープを確認する
5. **設計書更新** — 実装に先立ち、関連する設計書を更新してユーザーの承認を得る（詳細は「仕様駆動開発」セクション参照）
6. **計画提示** — `elaborate-issue` スキルで実装計画を策定し、ユーザーの承認を得る
7. 承認後に実装を開始する

#### 実装フロー

1. **ブランチ作成前に必ず main へ切り替え・最新化する**
   ```bash
   git checkout main && git pull
   ```
2. 最新の main からブランチ作成
3. PRでマージ（直接pushしない）
4. PR作成前に lint + test + build を通す
5. **設計書との最終同期** — 実装中に設計書から微修正が生じた場合、PR に設計書の更新も含める
6. **PRのレビューを GitHub PR Review として投稿する（PR作成後に必ず自動実施・確認不要）**
   - `gh pr review <PR番号> --comment --body "<レビュー内容>"` で投稿する
   - レビューコメントは以下の分類を使い分ける:
     - `[要修正]` — コード変更が必要な指摘（コメント追加・バグ修正等）
     - `[仕様差異]` — 設計書の記述と実装が一致しない箇所（設計書 or 実装の修正が必要）
     - `[補足]` — 問題なしの確認メモ。対応不要であることを明示する
   - `[good]` は変更不要の良い点の記録としてそのまま使用する
7. PRのマージは人間が確認・承認してから行う
8. **PR マージ確認後、作業ブランチをローカルから削除する**
   ```bash
   git checkout main && git pull
   git branch -d <branch-name>
   ```

### Issue/PRの状態確認

- Issue や PR の状態をユーザーに伝える際は、記憶に頼らず必ず `gh issue view` / `gh pr view` で最新状態を確認すること
- 作業記憶の情報は古くなっている可能性がある

## 失敗時の振り返りルール

- CI失敗、レビュー差し戻し、本番バグが発生したら **修正対応後に必ず** `retrospective` スキルを実行すること
- 修正完了 ≠ 対応完了。修正PR作成後、振り返りを完了するまでが対応フローである
- 振り返り結果は必ず `docs/learnings.md` に記録し、GitHub Issueも作成すること
- CLAUDE.mdのルール追加時は既存ルールとの整合性を確認すること

### GitHub Actions ワークフロー変更時のルール

- パーミッション設定は公式ドキュメントの推奨値と照合すること
- テンプレートから作成した場合、プロジェクト固有の要件（パーミッション、パス、シークレット名）を必ず確認すること

### レイアウト変更時のルール

- 既存コンポーネントをラッパー div で囲む前に、対象コンポーネントの `display` 特性（`inline-flex` 等）と親からのサイズ伝播を確認すること
- overlay/通知コンポーネントは `fixed` 配置を第一候補にし、既存レイアウト構造を変更しない
- レイアウトに影響する変更は `validate` に加え、ブラウザでの視覚的確認を行ってから push すること

### 後手（gote）表示ルール

- 後手の駒アイコンは外側 div に `style={{ transform: 'rotate(180deg)' }}` を適用して上下反転する
- Board・CapturedPieces など駒アイコンを表示する全コンポーネントで統一すること
- 新規コンポーネントで駒アイコンを表示する場合、`isSente` / `owner` に応じた回転を忘れずに実装する

### executeMove 後の currentPlayer 依存の禁止

- `executeMove` は呼び出しと同時に `currentPlayer` を次のプレイヤーへ切り替える
- 「誰が今の手を指したか」を取得したい場合は `currentPlayer` を使わず、`moveHistory.moves[currentIndex].piece.owner` を参照する
- `promotion_check` など `executeMove` 後のフェーズで表示する情報（ダイアログの駒色など）は移動履歴から取得する

## 仕様駆動開発

### 原則

- **設計書が実装の唯一の正（Single Source of Truth）である**
- 設計書に書かれていないことは実装しない。設計書と実装が乖離してはならない
- 新機能・仕様変更は「設計書更新 → 承認 → 実装」の順序で進める

### 設計書一覧と管掌範囲

| 設計書 | パス | 管掌範囲 |
|--------|------|---------|
| 要件定義書 | `docs/requirements.md` | 機能要件・非機能要件・技術スタック・モジュール構成 |
| 画面定義書 | `docs/screen-definition.md` | 画面レイアウト・コンポーネント一覧・インタラクション・表示仕様 |
| 状態遷移定義書 | `docs/state-transition.md` | GamePhase遷移・アニメーションチェーン・Undo/Redo・ヒント・ほめフロー |
| データ型定義書 | `docs/data-type-definition.md` | TypeScript型・ストアAPI・ロジックモジュールAPI・駒移動ルール |

### 設計書更新のタイミング

| タイミング | 対象 | 更新内容 |
|-----------|------|---------|
| 新機能のIssue実装前 | 関連する全設計書 | 新しい型・画面・状態遷移・インタラクションを追記 |
| 仕様変更のIssue実装前 | 変更影響のある設計書 | 既存の記述を修正 |
| バグ修正 | 通常は不要 | 設計書の記述自体が誤りだった場合のみ修正 |

### 設計書更新の手順

1. Issue の内容から、影響を受ける設計書を特定する
2. 設計書の該当セクションを更新する（型追加・画面定義追加・状態遷移追加等）
3. 更新差分をユーザーに提示し、承認を得る
4. 承認後、設計書の内容に基づいて実装計画を策定する

### 実装完了後の設計書同期

- 実装中に設計書の記述と異なる判断をした場合（値の微調整、API変更等）、PR作成前に設計書も更新する
- 設計書の更新は同じPR・同じブランチに含める（別PRにしない）
- PRレビューで `[仕様差異]` が検出された場合も、設計書 or 実装を修正して解消する

### 設計書の品質基準

- **曖昧性の排除**: AI が設計書だけを読んで実装できる具体性を持つこと
- **型定義との整合**: `data-type-definition.md` の型と `src/lib/shogi/types.ts` は常に一致すること
- **状態遷移の網羅**: 全ての GamePhase 遷移パスが `state-transition.md` に記載されていること
- **画面要素の網羅**: 全てのUIコンポーネントとそのpropsが `screen-definition.md` に記載されていること
- **情報の一元管理**: 同じ情報を複数の設計書に重複して書かない。正の所在を1箇所に決め、他は「→ 詳細は `<設計書名>` セクション X.X を参照」とする

### 情報の正の所在

| 情報カテゴリ | 正の設計書 | 他の設計書での扱い |
|------------|----------|-----------------|
| TypeScript 型定義（Piece, GameState, UIState 等） | `data-type-definition.md` | 参照リンクのみ |
| ストアAPI（GameStore のアクション一覧） | `data-type-definition.md` | 参照リンクのみ |
| 効果音・触覚の具体値（Hz, ms, パターン） | `data-type-definition.md` セクション 11.6, 11.7 | `requirements.md` は機能概要のみ。`state-transition.md` は SoundId 名のみ |
| ロジックモジュールAPI（関数シグネチャ） | `data-type-definition.md` セクション 11 | 参照リンクのみ |
| 画面レイアウト・コンポーネント仕様 | `screen-definition.md` | `requirements.md` は機能概要のみ |
| GamePhase 遷移・アニメーションチェーン | `state-transition.md` | 参照リンクのみ |
| 機能要件（何を実現するか） | `requirements.md` | 他の設計書は「どう実現するか」を記載 |

### 設計書変更時のコミットメッセージ

設計書を更新するコミットには以下を含めること:
- プレフィックス: `docs:`
- Issue番号: `(#XX)`
- 変更理由の要約

例: `docs: AI対戦の状態遷移・型定義を追加（#25）`

## Claude Codeとの協働指針

### 探索→設計→計画→実装→検証サイクル

複数ファイルにまたがる変更や新機能追加では、以下の順序で進める。

1. **探索**: 関連ファイルを読み、既存のパターン・設計を把握する
2. **設計**: 関連する設計書を更新し、ユーザーの承認を得る（仕様駆動開発セクション参照）
3. **計画**: 変更対象ファイルと実装方針を明確にする（`elaborate-issue` スキルを活用）
4. **実装**: 設計書と計画に沿ってコードを書く
5. **検証**: `validate` スキル（lint + test + build）を必ず実行してから PR を作成する

タイポ修正・1行変更など明らかに小さい修正はこのサイクルを省略してよい。

### コンテキスト管理

- 無関係なタスク間では `/clear` でコンテキストをリセットする
- 同じ問題の修正を2回以上試みても解決しない場合は `/clear` し、より具体的なプロンプトで再開する
- コードベースの広範な調査は subagent に委譲してメインコンテキストを温存する
  ```
  Use subagents to investigate how [機能] is implemented, then report findings.
  ```

### よくある失敗パターン（避けること）

- **無関係タスクの混在**: 1セッションで関係のない複数作業を行うとコンテキストが汚染される → タスク間で `/clear`
- **繰り返し修正**: 同じアプローチで2回失敗したら続けても悪化するだけ → `/clear` して学びを反映した新しいプロンプトで再開
- **検証なし出荷**: 動いているように見えても壊れているコードを出さない → 必ず `validate` スキルで lint + test + build を確認する
- **スコープなし探索**: 範囲を絞らずファイルを読ませるとコンテキストが枯渇する → 調査範囲を明示するか subagent に委譲する

## テストコード作成時の厳守事項

### テストコードの品質

- テストは必ず実際の機能を検証すること
- `expect(true).toBe(true)` のような意味のないアサーションは絶対に書かない
- 各テストケースは具体的な入力と期待される出力を検証すること
- モックは必要最小限に留め、実際の動作に近い形でテストすること

### ハードコーディングの禁止

- テストを通すためだけのハードコードは絶対に禁止
- 本番コードに `if(testMode)` のような条件分岐を入れない
- テスト用の特別な値（マジックナンバー）を本番コードに埋め込まない
- 環境変数や設定ファイルを使用して、テスト環境と本番環境を適切に分離すること

### テスト実装の原則

- テストが失敗する状態から始めること（red-green-refactor）
- 境界値、異常系、エラーケースも必ずテストすること
- カバレッジだけではなく、実際の品質を重視すること
- テストケース名は何をテストしているか明確に記述すること

### 実装前の確認

- 機能の仕様を正しく理解してからテストを書くこと
- 不明な点があれば、仮の実装ではなく、ユーザーに確認すること

## カスタムスキル一覧

| スキル                           | 用途                             | 実行タイミング                       |
| -------------------------------- | -------------------------------- | ------------------------------------ |
| `/elaborate-issue <Issue番号>` | Issue の実装計画策定             | Issue の実装指示を受けた時           |
| `/validate`                    | lint + test + build 一括検証     | PR 作成前（必須）                    |
| `/retrospective <説明>`        | 失敗の振り返り・学び記録         | CI失敗・差し戻し・バグ修正後（必須） |
| `/shogi-test <機能名>`         | 将棋ロジックのテスト設計・作成   | 将棋ロジックのテスト作成時           |
| `/sync-issue <Issue番号>`      | Issue のステータス・コメント更新 | 作業開始時・PR作成時                 |
