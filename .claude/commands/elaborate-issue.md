# elaborate-issue

Issue の実装計画を策定するスキル。

---
context: fork
argument-hint: "<Issue番号>"
---

## 手順

1. `gh issue view <Issue番号>` で Issue の内容を取得する
2. 以下の設計書を参照し、関連する仕様を把握する
   - `docs/requirements.md` — 要件定義
   - `docs/screen-definition.md` — 画面定義
   - `docs/state-transition.md` — 状態遷移
   - `docs/data-type-definition.md` — データ型定義
3. 既存コードを調査し、影響範囲を特定する
4. 以下のフォーマットで実装計画を出力する

## 出力フォーマット

```
## 概要
<Issue の目的と実現する機能の要約>

## 関連設計書
<参照した設計書のセクション番号を具体的に列挙する>
例:
- `requirements.md` セクション 3.2.6（ヒント機能）
- `screen-definition.md` セクション 2.2（対局画面）コンポーネント G-018
- `state-transition.md` セクション 2.5（ヒントフロー）
- `data-type-definition.md` セクション 5（UIState）、セクション 11.5（hint.ts API）

## 設計書の事前更新
<この Issue の実装に先立ち、設計書に追記・修正が必要な箇所を列挙する>
- 更新不要の場合は「更新不要」と明記する
- 更新が必要な場合は、どの設計書のどのセクションに何を追記/修正するかを具体的に書く

## 影響ファイル
<変更・新規作成するファイルの一覧と変更概要>

## 実装ステップ
1. <ステップ1>（根拠: <設計書名> セクション X.X）
2. <ステップ2>（根拠: <設計書名> セクション X.X）
...
※ 各ステップに、その実装の根拠となる設計書セクションを明記する

## テスト方針
<テストすべき観点・ケース>

## 懸念事項
<リスク・不明点・確認が必要な事項>
```

## 注意事項

- 計画出力後、ユーザーの承認を待つこと。勝手に実装を始めない
- 実装方針が複数ある場合は選択肢を提示する
- Issue の記述に曖昧な点があれば質問する
- 設計書と Issue に矛盾があれば指摘する
- **実装ステップは必ず設計書のセクション番号を根拠として引用すること**（設計書に記載がない実装は、まず設計書の更新が必要）
- 設計書に記載がない機能の実装が必要な場合、「設計書の事前更新」セクションに追記すべき内容を明記する
